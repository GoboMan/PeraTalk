## 収支シミュレーション（試算）

ここでの **手取り**は、アプリ／ストアから **開発者に入るキャッシュフロー**から **インフラ固定費と API 等の変動費（概算）** を引いた **粗い利益（月）** とする。**税・個人としての経費・確定申告・その他営業費は含めない**。

**目標：** その **手取りを月あたりおよそ ¥30,000** にするには、おおよそどの程度のユーザー規模が要るかを **1 シナリオ** で置く。

数字はすべて**仮置き**。単価・実装・為替で変わるので、四半期ごとなどで差し替える。

### 端的な結論（1シナリオ）

次の **ユーザー規模** を置いたときの、**月あたりの収支イメージ**です（**手取り ≈ ¥3万／月** に近づける例）。

| プラン | 月間アクティブ（例） |
|--------|----------------------|
| **無料**（広告あり） | **約 830** |
| **プラス**（広告あり・課金は月額のみと仮定） | **100** |
| **プロ**（広告なし・課金は月額のみと仮定） | **10** |

**月次キャッシュフロー（開発者側・概算）**

| 項目 | 金額 |
|------|------|
| **広告収入**（無料＋プラス **930** 人分。\(12\) 円／人・月） | **¥11,160** |
| **サブスク収入**（Apple **30%** 控除後。プラス¥500・プロ¥1,500 の**月額**のみ） | **¥45,500** |
| **収入合計** | **¥56,660** |
| **固定費**（インフラ等の代表値） | △ **約 ¥18,000** |
| **変動費**（API 等・次節の前提どおり） | △ **¥8,650** |
| **粗い利益（＝この試算での手取り）** | **¥30,010**（**およそ ¥3万／月**） |

※ 表では広告・固定・変動を **計算値どおり**示している。**粗い利益**は **¥30,010**（≒ ¥3万／月）。

**読み方：** **無料 MAU およそ830・プラス課金100・プロ課金10** くらいの規模を想定すると、**インフラと API を引いたあとの手元イメージは月およそ3万円**というオーダー。

なぜこの金額になるか・式は **次節**。

### 前提と計算式（詳細）

#### サブスク（開発者取り分）

- 店頭 **プラス ¥500／月**・**プロ ¥1,500／月** を **すべて月額** とし、Apple 手数料を **30%**（開発者取り分 **70%**）とする。
- プラス 1 人あたり月の取り分 \(r_{+} = 500 \times 0.7 = 350\)（円）。
- プロ 1 人あたり月の取り分 \(r_{\mathrm{pro}} = 1{,}500 \times 0.7 = 1{,}050\)（円）。

課金ユーザー数をそれぞれ \(N_{+}\)、\(N_{\mathrm{pro}}\) とすると、

\[
\text{サブスク収入（月）} = r_{+} N_{+} + r_{\mathrm{pro}} N_{\mathrm{pro}}.
\]

例では \(N_{+} = 100\)、\(N_{\mathrm{pro}} = 10\) なので

\[
350 \times 100 + 1{,}050 \times 10 = 35{,}000 + 10{,}500 = 45{,}500 \quad (\text{円}).
\]

※ 年額契約だけが増えたときは、**実効の月あたり単価**（¥250／¥1,000 相当）で \(r_{+}\)、\(r_{\mathrm{pro}}\) を置き換える。

#### 広告（開発者側・粗いモデル）

- **広告あり**は無料・プラスのみ。月間インプレッションを **1人あたり 60／月**（例：**2回／日 × 30日**）とする。
- **eCPM**（千インプあたりの広告収入の目安に使う指標）を **¥200**（中くらいの仮置き）とする。

\[
\text{広告収入（1人・月）} = \frac{60}{1000} \times \mathrm{eCPM} = 0.06 \times 200 = 12 \quad (\text{円}).
\]

広告ありユーザー数を \(N_{\mathrm{ad}} = N_{\text{無料}} + N_{+}\) と置く（プロは広告なし）、

\[
\text{広告収入（月）} = 12 \times N_{\mathrm{ad}}.
\]

例では \(N_{\text{無料}} = 830\)、\(N_{+} = 100\) なので \(N_{\mathrm{ad}} = 930\)、\(12 \times 930 = 11{,}160\)（円）。

※ メディエーション手数料や課税、eCPM の国差はここでは込みにしていない。**実測の eCPM・インプ数**に差し替える。

#### コスト

- **固定費** \(C_{\mathrm{fix}}\)：個人開発・早期の代表値として **¥18,000／月**（内訳は「音声モード別コスト試算」の固定費表を参照）。
- **変動費** \(C_{\mathrm{var}}\)：API・通信などを **プラン別の月あたり負担の代表値** \(c_{\text{無料}}, c_{+}, c_{\mathrm{pro}}\) で近似する。

\[
C_{\mathrm{var}} \approx c_{\text{無料}} N_{\text{無料}} + c_{+} N_{+} + c_{\mathrm{pro}} N_{\mathrm{pro}}.
\]

例では **テキスト中心・Live 音声は控えめ** の利用を想定し、\(c_{\text{無料}} = 5\)、\(c_{+} = 40\)、\(c_{\mathrm{pro}} = 50\)（円／人・月）と置いた。

\[
5 \times 830 + 40 \times 100 + 50 \times 10 = 4{,}150 + 4{,}000 + 500 = 8{,}650 \quad (\text{円}).
\]

**実際のトークン・音声秒数**が分かったら、[音声モード別コスト試算（参考）](#音声モード別コスト試算参考) の式で \(c\) を再見積もりする。

#### 利益（この試算での「手取り」）

\[
\text{粗い利益（月）} \approx \bigl(12 \cdot N_{\mathrm{ad}}\bigr) + \bigl(r_{+} N_{+} + r_{\mathrm{pro}} N_{\mathrm{pro}}\bigr) - C_{\mathrm{fix}} - C_{\mathrm{var}}.
\]

数値例では \(11{,}160 + 45{,}500 - 18{,}000 - 8{,}650 = 30{,}010\)（円）→ **およそ ¥3万／月**。\(c\) やインプ・課金構成が変わると、同じ手取りでも必要 MAU／課金人数は前後する。

## 音声モード別コスト試算（参考）

**位置づけ**

- **製品プラン**は上記どおり **無料（制限あり）／プラス・プロ（同一フル機能・差は広告のみ）**。**プラスとプロで B／C の開放など機能を分けない**。
- 以下の **Free／Standard／Max** は **API 原価試算用の利用パターン名**（無料ユーザーの軽利用／**課金ユーザーのフル機能**における B 寄り／C ヘビー等）。**Standard＝プラス、Max＝プロ**といった **製品対応は置かない**。数値はすべて**仮定**であり、リリース前に **実測トークン・実測クレジット消費**で置き換えること。

**利用パターン定義（試算用・案）**

| ラベル | 想定するユーザー | 主な内容 |
|--------|------------------|----------|
| **Free** | **無料プラン** | **広告あり**。**機能制限あり**（詳細は後から製品ドキュメントに記載）。試算上の例として **テキスト往復**を **1日おおよそ12〜18ラリー**程度に抑えるイメージを置く（実装は「クレジット」でも「ハード上限」でも可）。**B／C** は付けないか極小トライアルのみ（任意）。 |
| **Standard（B）** | **プラスまたはプロ**（同一機能）のうち **B（ターン制音声）を多用する**パターン | **機能はプラス・プロ共通**。広告の有無だけプラス／プロで異なる（収益は別節）。**音声クレジット**（例：**B 分**）やテキスト上限は **設計選択**の話であり、**プラスとプロで変える前提ではない**。 |
| **Max（C）** | **同上**のうち **C（リアルタイム／Live 級）を多用する**パターン | **機能はプラス・プロ共通**。**C 用クレジット**を**別レート**で消費するイメージ。原価が高いため **同梱分は控えめ**＋**追加クレジット課金（消費型）**の検討余地あり（実装・価格は別途）。 |

**コスト試算の共通仮定**

| 仮定 | 値 | メモ |
|------|-----|------|
| 為替 | **$1 = ¥150** | 変動で原価が動く。 |
| テキスト LLM | **Gemini 3.1 Flash-Lite 帯**の代表値として、**入力 $0.25／1M トークン**、**出力 $1.50／1M トークン** | [Gemini API 料金](https://ai.google.dev/gemini-api/docs/pricing) の最新表を優先。Vertex 利用時は同ファミリーの表示を確認。 |
| インタラクティブ1往復（テキスト）あたり | 入力 **2,500 トークン**（履歴・指示含む）、出力 **350 トークン** | 履歴を短く保つ実装なら下振れ。RAG・長文化で上振れ。 |
| セッション終了後の一括処理（ユーザーあたり月 N 回） | 入力 **6,000**、出力 **2,500**／回 | 総括＋ボキャブラリ候補。回数は下表で仮置き。 |
| **C（Live 級）の音声** | **入力音声 $3／1M トークン**、**出力音声 $12／1M トークン**、換算 **約25 トークン／秒** | [Vertex の Generative AI 料金](https://cloud.google.com/vertex-ai/generative-ai/pricing) の **Gemini 2.5 Flash Live API** 行などを参照。モデル・経路で変わる。**セッション請求の再計上**で実測は跳ねうるため **1.5〜3× の安全係数**を見込む余地あり。 |
| **B の音声** | **端末 ASR ＋ 上記テキスト LLM ＋ 端末 TTS** とし、**追加のクラウド音声課金はゼロ**（STT／TTS をクラウドにすると別途） | [会話](../機能/会話.md)・[会話-ペルソナとTTS](../機能/会話-ペルソナとTTS.md) の方針に沿う。 |

**変動費（API）のひとつの置き（「フルの人」ひとりあたり・月）**

| ユーザー像 | テキスト往復回数／月 | 終了後ブロック回数／月 | C 音声（壁時計分）／月 | 概算API（\$） | 概算API（¥, 150換算） |
|------------|----------------------|-------------------------|-------------------------|---------------|-------------------------|
| **Free（上限ギリギリ）** | **15／日 × 30 = 450** | 30（1日1セッション想定・仮） | 0 | 下の近似式で **約 \$0.56** | **約 ¥84** |
| **Standard「よく話す」** | 600 | 20 | **B 音声を多用**（例：月 180 分のセッション時間） | **端末 ASR** なら LLM は **テキスト往復**が主で、上式に **\(k=600\)、\(n=20\)** を代入した **約 \$0.80** 前後（**往復密度**を上げると比例）。**クラウド STT** なら **+\$数ドル／月** を見込む。 | **約 ¥120〜¥200**（STT なし〜ありの幅） |
| **Max「ヘビー」** | 800 | 30 | **C 360 分**（ユーザーと AI が半分ずつ話す想定の**壁時計**） | テキスト LLM **約 \$1.08**（\(k=800,n=30\) を上式に代入）＋ C 音声を**線形のみ**概算：約 **\$4〜5**。**セッション請求の再計上**などに **1.3〜2×** を掛けると **全体で \$7〜12／人・月** が一つのレンジ | **約 ¥1,050〜¥1,800**（為替150・**C が支配的**） |

式の内訳（参考）：

- 往復 \(k\) 回のテキスト LLM roughly:  
  \[
  \text{cost}_{\text{chat}} \approx \frac{k}{10^6}\bigl(2500 \times 0.25 + 350 \times 1.5\bigr) = k \times 0.00115\ \text{USD}
  \]
  \(k=450 \Rightarrow \text{約 }\$0.52\)。  
- 終了後処理 \(n\) 回:  
  \[
  \frac{n}{10^6}(6000 \times 0.25 + 2500 \times 1.5) = n \times 0.00525\ \text{USD}
  \]
  \(n=30 \Rightarrow \text{約 }\$0.16\)。合計 **Free 上限利用で ~\$0.56／人・月**。

**Standard（\(k=600,n=20\)）のテキスト LLM だけの目安**：\(600 \times 0.00115 + 20 \times 0.00525 \approx 0.795\) **USD**（**約 ¥119**）。ここに **クラウド STT** や **長い RAG コンテキスト**が乗ると上振れする。

**アップル手数料（簡易）**

- 初年度 **30%** → 開発者取り分 **70%**。  
- 小型事業者プログラム等で **15%** になったケースでは **85%**。  
- 以下の「製品プランとの対応」の表は **70% 取り分**を前提とした**店頭の箱**のイメージであり、**実際の利益**は **取り分 − API − 按分固定費**で再計算すること。

**製品プランとの対応（店頭価格は「プラン別機能一覧」の **プラス・プロ**列見出しセルが正）**

|----------------|-------------------------------|----------------|

**収支の読み方（例）**

- **固定費 ¥18,000／月** だけ見ると、**プラス月額の取り分 ¥350** なら **約 52 人**で固定費の「箱」だけ埋まる（API 無視の超概算）。**手取り ¥3万／月** を狙うときのユーザー規模は、上記の **収支シミュレーション** を正とする。  
- **実際**は無料ユーザー分の LLM と、**課金ユーザーのヘビー利用**が原価を引っ張るため、**(1) 無料の機能制限とクレジット上限 (2) 追加課金 (3) フェア利用・価格のバランス**を組むのが個人運営では安定しやすい。

**実装・運用上のチェックリスト**

1. **B／C は必ずサーバ側またはトークンで上限**（ユーザーID・日次・月次）。  
2. **クレジットの「1単位＝何秒／何往復」**を利用規約と UI で明示。  
3. **C の原価はログで追跡**（セッション秒・audio token・想定の安全係数）。  
4. **フル機能ユーザー（プラス／プロ）**が **C を使わない**月もあるため、**混合分布**で再シミュレーション。  
5. 料金は **[Gemini API](https://ai.google.dev/gemini-api/docs/pricing)** と **[Vertex](https://cloud.google.com/vertex-ai/generative-ai/pricing)** を**四半期ごと**に見直し。

**まとめ（本節）**

- **C を「無制限」で付ける**のはコスト的に難しく、**クレジットやフェア利用で上限**を設ける前提が現実的。
- **製品プランの正**は **無料＝制限あり／プラス・プロ＝同一フル機能（差は広告のみ）**。店頭は **プラン別機能一覧** の **プラス・プロ**列見出し。**Standard／Max** はこの節でのみ使う **原価試算用の利用パターン**。
- **固定費は早期で月 ¥数万以下に収めやすい**一方、**C の普及率が上がると変動費がボトルネック**になりやすい点だけは継続監視する。
