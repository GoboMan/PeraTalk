# 設定とアカウント

[← 機能一覧に戻る](機能一覧.md) ／ [← README に戻る](../../README.md)

ログ／インプット／アウトプット以外で、**設定画面を中心**に提供する機能群（ファーストフェーズに含める）。

---

## 目的・ユーザー価値

- 利用に必要な**アカウント運用**・**プラン確認**・**外観の好み**・**法務文書へのアクセス**・**レビュー導線**を 1 か所に集約する。

---

## 1. 仕様

| 項目 | 内容 | 備考 |
|------|------|------|
| **アカウント（登録／編集）** | **サインアップ**、**ログイン**、**ログアウト**、および**プロフィール等の編集**（表示名・連絡先など。**載せる項目は実装で確定**）。初回利用時のセットアップを含む。 | 認証方式（例：Sign in with Apple、メール）は別途決定。**機種変更時のデータ引き継ぎ**：Phase 1 と Phase 2 で復元できる範囲が異なる（Phase 1 はカレンダー・総括・単語帳まで／Phase 2 で発話本文も復元）。境界・候補スナップショット非永続を含む詳細は [会話 §5](会話.md#5-データの保持削除長期記憶rag) が正。アカウント削除時は **サーバー側の全データ**を CASCADE で消し（[データベース設計-サーバー §1.3](../アーキテクチャ/データベース設計-サーバー.md#13-アカウント削除フロー)）、端末データはアプリ／OS の削除フローに従う旨をプライバシーポリシーで案内する。 |
| **サインアウト・アプリ内データを消去（確認ダイアログ）** | **サインアウト**および**「アプリ内データを消去」**を実行する前の**確認ダイアログ**で、**Phase 1 期間中は会話の発話本文がこの端末にしか存在しない**旨を必ず明示する。「**この端末を手放した／壊れた／消去した場合、Phase 1 期間中の発話本文は復元できません**（カレンダー・総括・単語帳はサーバーから復元されます）」のような文言で警告する。**Phase 2 リリース後はこの警告を撤去**し、リリース告知で「旧端末を一度起動すれば過去発話も自動でサーバーに同期される」を案内する。 | 構造的制約の根拠と Phase 2 の解消経路は [データベース設計-クライアント §3.2](../アーキテクチャ/データベース設計-クライアント.md#32-cachedutterance--セッション内の発話-1-件phase-1-は端末のみphase-2-で双方向同期) ／ [会話 §5](会話.md#5-データの保持削除長期記憶rag)。アカウント削除（全データ削除）は本ダイアログとは別フローで、サーバー側を CASCADE で消すため Phase に関係なく不可逆。 |
| **データ寿命ポリシー（長期非ログイン）** | **長期間ログインがないユーザーのデータを段階的に削除する方針**を持つ（GDPR・個人情報保護法の「目的達成後は遅滞なく削除」原則と整合）。**実装は Phase 2 以降**で、Phase 1 ではポリシーの存在をプライバシーポリシーに明記し、`profiles.last_seen_at` の値だけ更新しておく。 | 詳細は [データベース設計-サーバー §1.4](../アーキテクチャ/データベース設計-サーバー.md#14-データ寿命ポリシー長期非ログイン時の自動削除)。概要：**6 ヶ月で警告メール 1 通目／12 ヶ月で発話本文の自動削除（カレンダー・総括は残す）／24 ヶ月でアカウント全体削除**。プラン共通（無料／プラス／プロで差を付けない。プラン別の保管期間差別化は将来検討）。 |
| **プラン** | **無料／プラス／プロ（サブスク）**の確認と、**課金・復元**への導線（StoreKit）。 | 価格・機能の正は [収益とサブスク](../運営/収益とサブスク.md) のプラン別機能一覧と一致させる。 |
| **テーマ／外観** | **テーマカラー**（例：アクセント色）や、**ライト／ダーク**に追従するなど表示の好み。 | 具体的なプリセット数は実装で確定。 |
| **利用規約** | アプリ内または Safari で**利用規約**を表示。 | 文言・ホスティングはリリース前に確定。 |
| **プライバシーポリシー** | 同上で**プライバシーポリシー**を表示。 | 広告・分析を入れる場合は必須に近い。 |
| **レビュー** | **App Store レビュー**への導線を二段構えにする。<br>（1）**適切なタイミング**でシステムのレビュー依頼（`requestReview` 等）。<br>（2）**設定からいつでも**「評価する」でストアレビュー画面またはストアの該当ページを開ける。 | 依頼の頻度はガイドラインと UX を踏まえて制御。設定経路で常に用意し、実装コストも低めに載せられるようにする。 |
