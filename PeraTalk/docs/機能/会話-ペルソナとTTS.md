# 会話-ペルソナと TTS

[← 機能一覧に戻る](機能一覧.md) ／ [← README に戻る](../../README.md)

AI 会話で「仮想の相手」を表現するための **Persona（ペルソナ）** と、**読み上げ（TTS）** の方針をまとめる。会話ループ本体は [会話](会話.md)、LLM 推論は [LLM-API方針](../アーキテクチャ/LLM-API方針.md)、キャラ声の高度な将来案は [将来-キャラ音声](../ロードマップ/将来-キャラ音声.md) を参照。

> **音声入力のスコープ（A／B／C）**は [音声入力フェーズ](../アーキテクチャ/音声入力フェーズ.md) を参照（本ドキュメントでは再掲しない）。**C の仕組み・料金**は [将来-リアルタイム音声](../ロードマップ/将来-リアルタイム音声.md)。

---

## 1. 用語

- **Persona（ペルソナ）**  
  表示名・声（アクセント／性別）・（必要なら）プロンプト上の口調など、**「誰と話しているか」をアプリが表現するための設定の束**。機械学習の **LLM モデル名**（例：Gemini のモデル）とは別概念とする。コードや DB では `Persona` / `ConversationPersona` などと名付けると混乱が減る。

- **読み上げ（TTS）**  
  LLM が返した**テキストを音声にする**処理。初版は **iOS のオンデバイス合成**（例：`AVSpeechSynthesizer`）を主軸とする想定。

---

## 2. TTS 技術選定（オンデバイス vs クラウド）

> 単語帳の読み上げ（[単語帳](単語帳.md) `VOC-LISTEN`）も同じ技術選定の対象。会話側と方針を揃えるため、本節は**会話と単語帳の双方からの単一参照点**として置く。

| 方式 | アプリ／運用での「読むたび」の追加課金 | メモ |
|------|----------------------------------------|------|
| **オンデバイス TTS**（Apple 標準など） | **音声 API の従量課金は発生しにくい**（合成は端末内。ストア配布のための開発者プログラム等は別） | 品質は「OS が提供する声」に依存。利用可能な声は端末・ユーザーが DL した高品質音声などで差が出る。 |
| **クラウド TTS**（各種ベンダー） | **文字数・リクエストに応じた課金**が一般的。多くは**月次の無料枠**のあと有料 | 高品質・バリエーションは有利だが、**超過時の料金設計**と監視が必要。**音声ファイルをサーバーへ永続保存しない**前提は [インフラ-Supabase](../アーキテクチャ/インフラ-Supabase.md) を参照。 |

**LLM 費用**（トークン課金）と **TTS** は別ライン。初版で TTS をオンデバイスに寄せれば、**サーバー側の音声合成コストは設計上ゼロ**にしやすい。

---

## 3. 初版のペルソナ数と軸

**男性／女性 × アメリカ英語／イギリス英語／オーストラリア英語**の **6 パターン**を、初回リリース時点の **固定ラインナップ**として定義する。

- **アクセント／ロケール**  
  読み上げおよび（必要なら）プロンプト上の口調の基準として `en-US` / `en-GB` / `en-AU` を使い分ける。

- **性別**  
  利用可能な **システム音声**のうち、意図する性別に最も近い声を選ぶ（端末に声が無い場合は **フォールバック** を定義する）。

**表示名の例**（最終コピーはリリース前に確定でよい）：

| 軸 | 表示名の例（案） | メモ |
|----|------------------|------|
| アメリカ・男性 | **Bob** | ドキュメント上の例示名 |
| イギリス・男性 | **Michael** | ドキュメント上の例示名 |
| オーストラリア・男性 | （リリース前に確定） | — |
| アメリカ・女性 | （リリース前に確定） | — |
| イギリス・女性 | （リリース前に確定） | — |
| オーストラリア・女性 | （リリース前に確定） | — |

「人間らしい記憶」やパートナー単位の文脈分離の考え方は [コンセプト-人間らしい記憶](../概要/コンセプト-人間らしい記憶.md) と整合させる（**ペルソナ／パートナー単位**で記憶の見え方を分ける想定と相性がよい）。

## 4. 定義の置き場所（初版 → 将来）

### 初回リリース：アプリ内定義

- **6 人分の Persona** は **アプリバンドル内**（Swift の静的データ、または同梱 JSON 等）に持つ。
- メリット：オフラインで一覧が確定する、実装が単純、**ストア更新なしに増やせない**デメリットは初版では許容しやすい。

### 将来：データベースのマスター化

人数増加・**リリースなしでの追加・無効化**・課金との連動などが必要になったら、**Supabase 等のマスターテーブル**に移す（または **リモート＋オンデバイスキャッシュのハイブリッド**）を検討する。

**後から移行しやすくするための前提**（初版のデータ設計で意識しておく）：

1. **安定した `slug`（例：`us-male-bob`）**を必ず持ち、表示名は変更可能にする。  
2. **Persona 一覧の取得を 1 か所に集約**し、UI はその抽象にだけ依存する（将来、ネットワーク取得に差し替えやすい）。  
3. **声の指定**は、OS の **voice identifier の優先リスト＋ロケール／性別フォールバック**など、DB の行に写しやすい形で持つ。

---

## 5. 品質・ナチュラルさの位置づけ

- **標準オンデバイス TTS**は、**追加のクラウド費用なし**で **6 パターンの差別化**には十分使える一方、**「本物の人間との通話」級**の表現力は期待しすぎない。  
- 不満が出た段階で、**クラウド TTS** や **OSS オンデバイスモデル**、将来の [将来-キャラ音声](../ロードマップ/将来-キャラ音声.md) に振り分ける。
