# LLM-API 方針（英会話・将来の多言語拡張）

[← README に戻る](../../README.md)

> 本ドキュメントは **クラウド LLM（Gemini）のプロバイダ・既定モデル・呼び出し経路の単一ソース**。**オンデバイス推論との役割分担**もここで定義する。単語帳の UI・ドラフト仕様の詳細は [単語帳](../機能/単語帳.md) §5。他ドキュメント（README／会話／インフラなど）は**本書を参照**するに留める。**単語帳まわりのオンデバイス生成は Apple Intelligence が提供する端末内モデルを使い、アプリがオープンモデル等を別途ダウンロードして載せる方針は取らない**。

---

## 推論の役割分担（クラウド vs オンデバイス）

| 経路 | 用途 | 狙い |
|------|------|------|
| **クラウド（Gemini・生成）** | **英会話**の相手発話・文脈維持、**セッション終了後**の講評・**ボキャブラリ候補**の抽出など、表現の幅と文脈が効く処理 | 品質と**構造化出力**（Structured outputs）を API 側で統一。キー・課金はバックエンド集約（下表）。 |
| **クラウド（Google・埋め込み）** | **記憶用セッション要約**（[会話 §5](../機能/会話.md#5-データの保持削除長期記憶rag)）のベクトル化。pgvector への投入元 | **`text-embedding-004`（768 次元・安定版）**を既定とする。次元固定型のため、変更時は全ベクトル再生成＋スキーマ変更が必要（[データベース設計-サーバー §3.14](データベース設計-サーバー.md#314-session_memory_chunks--要約のベクトル1-セッションあたり-13-行)）。 |
| **オンデバイス（Apple Intelligence）** | [単語帳](../機能/単語帳.md) の**単語追加／編集**で使う **AI 一括ドラフト**（**英語定義**・補助語の説明・**例文**・用法などの下書き） | **Gemini API の呼び出しを原則発生させない**。単語操作は回数が増えやすいため従量課金と通信を抑える。処理は **Apple が OS とともに提供するオンデバイスの言語モデル**に依存し、アプリ側で **別モデルをダウンロードして保持する構造は不要**。幻覚・品質はユーザー編集・保存前ドラフトで吸収する前提。 |

- **実装経路**：Apple が公開する **開発者向け API**（例：**Foundation Models** ほか、リリース時点の公式ガイドに従う）でオンデバイス生成を行う。**対応ハードウェア・OS バージョン・ユーザーによる Apple Intelligence の設定**により利用可否が変わるため、**非対応時の UX**（クラウドへの限定フォールバック／機能を無効化／手入力のみ等）は実装・プロダクトで確定する。
- **例外**：オンデバイスが利用不能な環境のみ **Gemini 等クラウドへ逃がす**選択肢はありうる。その場合は**既定はオンデバイス**とし、通信・課金・プライバシー表示を分けて整理する。

### 会話とオンデバイスの相乗効果（設計意図）

- **クラウド（会話）**：ユーザーが**学習・記録対象にしている語彙**（単語帳のエントリ・セッション後の候補など）を、プロンプトへ**背景情報として注入**できる。相手役がその語を**自然に織り込んで話す**、ユーザーが使った際に**短い肯定フィードバック**を返す、などは**実装で具体化**する（教育的ねらいは [ビジョンと目的](../概要/ビジョンと目的.md)）。
- **オンデバイス（Apple Intelligence）**：単語帳の一括ドラフト（[単語帳](../機能/単語帳.md) §5）に加え、**記録直後**に文脈に即した意味や**ユーザー向けの別例文**を端末内だけで出すフローを載せられる。**通信待ち**と **Gemini の従量**を抑えつつ、**単語ドラフトまわりのテキストをクラウドに送らない**選択肢を取りやすい（対応端末・設定時）。

---

本アプリでは、上表のとおり**会話まわりの推論**を **Google** の **Gemini** で行う。**初版**は会話・フィードバックの**主言語を英語**にそろえ、プロンプトと評価の負荷を抑える。**将来的に**他言語の学習ペアを足す場合も、同じ API 枠・テンプレ構造を流用する想定。

**呼び出し経路（クラウド）**：開発・試行・キー発行は **Google AI Studio**（**Gemini API**）を前提とする。**本番**では、請求・データ所在地・運用ポリシーに応じて **Vertex AI**（**Google Cloud** 上の Gemini）へ寄せる選択肢がある（同一 Gemini ファミリーの利用が一般形）。

**会話をクラウドに寄せる**ことで、会話往復ごとに**大規模モデルをアプリが配布する必要はない**。**単語帳の AI ドラフト**は **Apple Intelligence のオンデバイスモデル**を利用するため、**オープンモデルの同梱・オンデマンド DL に伴うストレージ・初回セットアップ**は設計対象から外す（対応端末では OS・Apple の提供形に依存。レイテンシ・発熱・バッテリは実機で検証）。インフラ側のデータコストとの関係は [インフラ-Supabase](インフラ-Supabase.md)。クラウド経路では**コスト**と**構造化出力**を重視した方針を以下にまとめる。

## 0. 製品方針（既定・拡張）

| 項目 | 方針 |
|------|------|
| **プロバイダ（クラウド）** | **Gemini**（**Google**）。利用開始・試行は **Google AI Studio**（**Gemini API**）。本番は **Vertex AI**（**Google Cloud**）で同系モデルを呼ぶ構成も含めて検討する（**API キーはモバイルに埋め込まず**バックエンド／BFF に限定）。 |
| **単語帳の AI ドラフト** | **オンデバイス（Apple Intelligence）**を既定とし、**原則 Gemini API を呼ばない**（役割分担は本章冒頭の表）。 |
| **オンデバイス側のモデル** | **Apple が提供するオンデバイス言語モデル**（実装は OS が用意する API に委ねる）。アプリが **ユーザーにモデルを選ばせない**。ウェイトの配布・バージョン管理は **Apple／OS の更新**に追従する。 |
| **既定モデル（クラウド・生成）** | **Gemini 3.1 Flash Lite**。API 上のモデルコードは現時点で **`gemini-3.1-flash-lite-preview`**（プレビュー。GA や名称変更時は [モデル一覧](https://ai.google.dev/gemini-api/docs/models/gemini-3.1-flash-lite-preview) を優先して本書を更新する）。 |
| **既定モデル（クラウド・埋め込み）** | **`text-embedding-004`**（Google・**768 次元**・安定版）。**記憶用セッション要約**のベクトル化に使う。pgvector の `vector(768)` と一致させ、モデル変更時は**全ベクトル再生成＋スキーマ変更**が必要（[データベース設計-サーバー §3.14](データベース設計-サーバー.md#314-session_memory_chunks--要約のベクトル1-セッションあたり-13-行)）。 |
| **クライアントからの直叩き** | **API キーはモバイルアプリに埋め込まない。** バックエンド（または BFF）が代理で呼び出し、認証・レート制限・利用量の集約を行う想定。 |
| **構造化出力** | セッション終了後の **総括フィードバック・ボキャブラリ候補** などは、Gemini API の **Structured outputs**（`response_json_schema` 等）でスキーマに沿わせ、パースと検証コストを抑える（失敗時のリトライや短いフォールバックは実装で定義）。 |
| **ユーザーからのモデル指定** | **不可**。クライアント（設定画面を含む）から **Gemini のモデル ID を選ばせない**。**Apple Intelligence 側も**ユーザーが任意のチェックポイントへ差し替えることは想定しない（OS が提供する範囲に従う）。クラウド側の実際に呼び出すモデル（**Flash Lite** のまま／**Flash** や **Pro** へ変更／プレビューから安定版への移行など）は**バックエンド・運用設定およびデプロイのみ**で決める。会話の文脈・終了後処理の流れは [会話](../機能/会話.md) を参照。 |

## 1. 想定している負荷（プロダクト由来）

| 用途 | 内容 | API 利用上のポイント |
|------|------|----------------------|
| **インタラクティブ** | **初版は英語**での相手役の発話、短い文脈の維持 | **往復ごとのリクエスト数**・レイテンシ。ストリーミングは UX 次第で検討。 |
| **教育系フィードバック** | 誤り指摘、言い換え、場面に合う表現 | 指示テンプレの一貫性。**英語（会話）**と**説明・補助言語**（例：日本語）を明示。 |
| **セッション後処理** | **総括**の講評・**ボキャブラリ候補**など、**構造化されやすい長めの出力** | **出力トークン量**が課金に効く。スキーマで無駄な前置きを減らす。 |
| **単語帳：AI 一括ドラフト** | [単語帳](../機能/単語帳.md) §5 の **AI生成**（見出し語から定義・例文・用法などを下書き） | **既定では Gemini API 対象外**（**Apple Intelligence オンデバイス**）。推論時間・発熱・対応端末は実機検証と OS の仕様に依存。 |
| **将来：蓄積に基づく追加フィードバック** | 蓄積された会話・総括・ログ等を入力に、ユーザー依頼で**追加の講評や傾向**を返す | **入力コンテキストが伸びやすい**。ウィンドウ・要約・キャッシュ（利用可能なら）と課金試算が必要。詳細は [会話](../機能/会話.md) の将来検討を参照。 |
| **将来：多言語拡張** | 学習言語を英語以外にも広げる場合 | プロンプトで言語を固定し、**回帰用の短い評価セット**を言語別に持つ（初版は英語セットから開始）。 |

## 2. 選定の軸（優先順位の目安）

1. **コスト**：**Flash Lite** 帯を既定にし、プロンプト長・会話履歴の **ウィンドウ管理**（要約・トリム）と **コンテキストキャッシュ**（利用可能）でトークンを抑える。
2. **構造化出力**：フィードバック系は **スキーマ付き応答** を優先する（[モデル仕様上 Structured outputs 対応](https://ai.google.dev/gemini-api/docs/models/gemini-3.1-flash-lite-preview)）。
3. **プライバシー・運用**：送信データの範囲（保持する会話ログの粒度）、**データ保持ポリシー**・利用地域は **Google AI Studio（Gemini API）** または **Vertex AI** のいずれを採用するかに応じた利用条件・プロダクトのプライバシーポリシーと揃えて文書化する（詳細は実装フェーズで確定）。**サーバー DB のうち RAG に載せる会話由来テキストは記憶用要約（サニタイズ済み）のみ**とする。**発話本文は Phase 1 は端末のみ／Phase 2 は `session_utterances` に載せるがベクトル化対象にはしない**、という境界は [会話](../機能/会話.md) §5 が正。

## 3. プロンプトと実装のプラクティス

- **テンプレ**：**初版は「英語＋説明・補助言語」＋難易度**を**毎回明示**する。将来、学習言語が増えても**同じ枠**に載せる。モデル ID を変えてもテンプレ構造は揃える。
- **後処理**：候補の **Kind** はスキーマに近い形で返す。**パース失敗**時はリトライや短いフォールバックを設計する。
- **Thinking（該当モデル時）**：AI Studio／API で思考量を調整できる場合は、**高頻度チャットは低め**、**終了後のまとめ生成は必要に応じて上げる**など、コストと品質のバランスを実測で決める。
- **監査**：開発時はログに **リクエスト／トレース ID** を残し、不具合調査とコスト突合しやすくする。
