# コンセプト-人間らしい記憶（AIらしさを抑える「記憶」とパートナー単位の文脈）

[← README に戻る](../../README.md)

このドキュメントは、[ビジョンと目的](ビジョンと目的.md) の「コンセプト・差別化」で触れた内容を、意図・ユーザー体験・実装の筋道まで掘り下げて書いたものです。

---

## ねらい（プロダクト全体）

- **AIであることを意識させない**方向で英会話体験を設計する（テキストの体裁・応答の機械臭・「すべてを把握している万能感」の過剰さなどを抑える）。
- その一方で、**心理的なハードル**は下げたい（堅苦しいインタビュー感や、何度も同じ説明を強いられる疲れを減らす）。
- 英語学習としては、**自分の状況を言葉にして説明する機会**を残したい。相手が勝手にすべてを「知っている」ように見えると、説明のアウトプットが減り、練習にならない。
- **モチベーションと継続**：自分の発話や背景が「ちゃんと相手に残っている」ように感じられると、関係性へのポジティブな感情や**また話したい**という動機が生まれやすい。その心理的報酬は**対話の回数・長さ**に効き、英語学習ではそのまま**アウトプットの量・継続**につながることが期待できる。だから「記憶」の見せ方と RAG は、体験品質だけでなく**学習者的モチベーションの設計**としても位置づける。

---

## 表面的な「記憶」と裏側の仕組み（RAG）

**ユーザーに見せたいこと**

- 同じ AI パートナーと話しているとき、**前に話したことが続いている**ように感じられる（毎回ゼロから自己紹介し直すゲームにならない）。

**裏側の考え方**

- 全会話を無制限にコンテキストwindow に載せるのではなく、**セッション単位のサニタイズ済み要約をベクトル化して保存**し、応答生成のたびに**必要な分だけ検索（retrieve）**してプロンプトへ**合成（compose）**する（いわゆる RAG：Retrieval-Augmented Generation）。
- 「すべてを丸ごと覚えている」モデルではなく、**都度、関連する断片を引いてから話している**という実装にはなるが、UX 上は**連続した対話の記憶**として見えることを優先する。

**これで解きたい苦痛**

- **同じ相手**に、同じ背景説明や同じ質問を何度も繰り返すことへのストレスを減らす（現実の友人とはまた質感が違うが、「続きの会話」として自然な範囲に寄せる）。

## 記憶の期待値と長期データの置き方（方針）

- UX 上の期待値は **話題の大枠が続いているように感じられればよい**レベルとする。**一言一句の再現は目的としない**。
- 長期の retrieve（RAG）用には、**セッションごとに 1 本に濃縮した「記憶用セッション要約」**を用意し、**それをベクトル化**する。**RAG に入るテキストは常に、この要約（サニタイズ済み・サーバー保存済み）に限定**する。**発話原文をそのままベクトル化しない**。**ラリーごとの要約を個別にベクトル化する**方針は当面取らない（必要になれば別途検討）。
- **発話単位の本文（会話本文の時系列）は Phase 1 は主に端末に保持**し、**Phase 2 からサーバー（`session_utterances`）へも同期する**（詳細・総括の境界は [会話](../機能/会話.md) §5 が正）。**記憶用セッション要約**はルールサニタイズと LLM 要約のパイプライン経由でサーバーに載せる。**総括フィードバック**も Phase 1 からサーバー同期する。**ベクトル**は要約由来でサーバー側に置く。
- ベクトルストアには **要約テキスト（長い場合は意味単位で少数チャンク）**が載る。**要約／ベクトルの削除ポリシー**とアカウント削除時の一括消去は [会話](../機能/会話.md)・[インフラ-Supabase](../アーキテクチャ/インフラ-Supabase.md) と整合させる。

### 発話単位ベクトルとの差を埋めるためのコツ（要約 1 本運用）

ラリーごとにベクトルを載せない代わりに、**記憶用セッション要約の中身と検索のかけ方**で「細かさ」をなるべく補う。

- **要約は自由文 1 段落だけにしない**：生成時に **決めたスキーマ**で書かせると検索・合成で埋もれにくい。例（案）：**話題（Topics）**／**ユーザーが伝えた事実・決めたこと（Facts）**／**練習した・出てきた英語フレーズ（Phrases）**／**次に続きそうな論点（Open threads）**。英語学習アプリなら **Phrases** と **Facts** が特に効く。
- **総括フィードバックと役割を分ける**：総括は「講評・学習」、記憶用は「あとから検索される **パートナーとの共有メモ**」として**情報密度を高く**（感情や冗長な講評より、**何について話したか**が残るように）。
- **retrieve は 1 本だけ引く前提にしない**：同一ペルソナ内で **類似度上位 k 件**（複数セッションの要約）をプロンプトに載せると、「だいぶ前の別の日の京都の話」との橋がかかりやすい。**メタデータ**（日付・ペルソナ slug）で絞ってからベクトル検索してもよい。
- **長いセッションは要約を 2〜3 チャンクに分割**（話題が切り替わる単位）。ベクトルもチャンク単位にすると、「ホテルの話」と「チェックインのフレーズ練習」が**別チャンクとして検索ヒット**しやすくなる（[会話](../機能/会話.md) のチャンク方針と整合）。
- **retrieve が弱いときの振る舞い**：プロンプト側で「検索結果に無いことは断定しない／聞き返す」を守ると、**無理に覚えたフリ**によるズレを減らせる（体験上の信頼と両立）。
- **（任意・将来）ユーザー明示の「覚えておいて」**：ピン留めや単語帳連携で **フレーズだけ別行**として要約や別ストアに載せると、**全文の発話ログ無しでも**表現の想起を底上げできる。初期実装の範囲外でもよいが、差分を埋める lever として覚えておく。

## パートナー（人）単位で記憶を分ける

**想定する体験**

- ユーザーが複数の AI パートナー（例：A・B・C）を持つとき、**A とはよく話すが B・C とはあまり話さない**といった偏りがあってよい。
- **B と話したセッションでは、A にだけ話していた内容は知られていない**ように見せる。現実で「まだその話をしていない相手」には、普通は説明が必要なのと同じ論理である。

**設計上の含意**

- 記憶ストア（ベクトルインデックスやメタデータ）は、少なくとも **`ユーザー × AIパートナー` の単位**で論理的に分離する（実装は単一インデックス + メタデータフィルタでも、パートナーごとのコレクションでもよい）。
- クロスパートナーで情報を共有しないことで、「説明しなくていい相手」による**産出のすり抜け**を避けつつ、**パートナー内では RAG で繰り返し負担を下げる**、という二段構えになる。

## トレードオフと境界（メモ）

- **見せ方と実装のギャップ**：ユーザーには「覚えている」と見えるが、検索が失敗したり関連が薄いと**忘れたように見える瞬間**はありうる。どこまでを許容し、UI でどう補うかは別途検討余地がある。
- **プライバシー・データ**：**Phase 1 は発話本文が端末中心／Phase 2 はサーバーにも載る**。それでも **RAG に載せるのはサニタイズ済み要約のみ**で、**発話原文をそのままベクトル化しない**ことなどの境界は [会話](../機能/会話.md) が正とする。それでも **LLM 処理中の委託先**や **端末紛失**など残リスクは説明が必要。削除・エクスポートは同ドキュメントおよび [インフラ-Supabase](../アーキテクチャ/インフラ-Supabase.md) と整合させる。
- **プロンプト設計**：「知っているフリをしすぎない」「知らないときは聞き返す」など、人格・シーンに応じたガードレールが RAG と両立するよう調整する。
