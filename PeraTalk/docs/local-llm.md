# ローカル LLM の選定（多言語語学用途）

[← README に戻る](../README.md)

本アプリでは推論（会話相手・文法・言い換え・セッション後のフィードバック／ボキャブラリ候補の生成など）を**主にオンデバイス**で行う前提である（ [データ量・Supabase](infrastructure-supabase.md) ）。ここでは **学習言語が英語に限らない** ことに加え、**どの端末でもまず動かす**・**必要な人だけ追加資産を入れる**という製品方針をまとめる。

---

## 0. 製品方針（既定・追加・切替）

| 項目 | 方針 |
|------|------|
| **既定（バンドル）** | **Gemma 3** のうち、**できるだけ軽量で、幅広い端末で動作させる**候補（具体のサブバリアント・量子化は実装・実測で確定。例：**1B 級 Instruct**）。初回ダウンロードサイズと下限端末での安定性を優先する。 |
| **追加ダウンロード** | ユーザーが**オプションで取得**する追加パックを用意する。**高品質・多言語で有利になりやすい系統**（例：**Qwen** の小型〜中型 **Instruct**）を第一候補として置く。**採用する型番・ファイル形式**は実装フェーズで更新する。 |
| **モデルの切替** | **インストール済みのモデルのうち、ユーザーが「今使うモデル」を選べる。** 設定から変更できる想定に加え、**会話セッションの途中でも切り替え可能**とする（詳細な挙動は [Conversation フロー](flow-conversation.md) の「モデル切替」）。 |
| **ストレージ** | 既定はアプリに同梱または最小取得。**追加モデルは端末ローカルに保存**し、Supabase 試算の「テキスト DB」とは別に、**ユーザー端末の空き容量**を圧迫しうる旨を設定やストア説明で考慮する（下記「複数モデル時の容量」）。 |

**複数モデル時の容量（目安）**

- **両方（既定＋追加）を保持すると、おおよそ「各モデルファイルの合算」分だけアプリ占有が増える。** 量子化 **Q4 前後**の例では、**〜1B 級で約 0.7〜1.2 GB／本**、**〜3〜4B 級で約 1.5〜3 GB弱／本**程度と見ておくとよい（**正式数値は採用する GGUF／MLX と量子化で決まる**）。
- そのため **合計で 2〜4 GB 級**に届くことは十分あり得る。設定で **追加モデルの削除（クリーンアップ）**・**ダウンロード前の概算表示**・**Wi‑Fi 推奨**などを用意すると、ユーザー期待と実機ストレージの齟齬を抑えられる。

## 1. 想定している負荷（プロダクト由来）

| 用途 | 内容 | LLM に求めやすい性質 |
|------|------|----------------------|
| **インタラクティブ** | 学習言語での相手役の発話、短い文脈の維持 | レイテンシ、同言語での自然な応答 |
| **教育系フィードバック** | 誤り指摘、言い換え、場面に合う表現 | 指示追従、**学習言語＋説明用の母国語（または学習言語のみ）**の切り替え指示への追従 |
| **セッション後処理** | 発話単位・総括の講評、**構造化されやすい長めの出力** | **長いコンテキスト**、JSON／箇条書きなど形式指定の安定性 |
| **多言語** | 英語・韓国語・スペイン語など、ユーザーの**学習言語ペア**ごとに同等に扱いたい | 言語カバレッジ（特に**日中韓**とヨーロッパ言語のバランス） |

**2 段階モデル**にすると、**既定の Gemma 3（軽量）は**バッテリー・初回体験に有利だが、**終了後処理や CJK 中心の学習**では**オプション（例：Qwen）の方が安定しやすい**ことがありうる。UI では「**品質重視は追加モデル推奨**」のように期待値を置けるとよい。

---

## 2. 選定の軸（優先順位の目安）

1. **既定モデル**：**推論遅延・メモリ・発熱**の最坏ケース（想定下限端末）でセッションが破綻しないことを最優先。多言語は「すべて最高品質」より**まず通る**ことを優先。
2. **オプションモデル**：**多言語・指示追従・長文後処理**の欠けを埋める。**Qwen 系小型 Instruct**はこのスロットの第一候補（実測で確定）。
3. **切替時の一貫性**：プロンプトテンプレは**モデル共通**に寄せ、差はパラメータ（温度・最大トークン等）と**実装上限**で吸収する。
4. **ライセンス**：同梱／オンデマンド配布の両方で、利用条件を確認する。

---

## 3. 既定とオプションの役割分担（目安）

| 区分 | 想定する系統 | 役割のイメージ |
|------|----------------|----------------|
| **既定** | **Gemma 3**（最軽量帯・Instruct） | 全端末での**会話ループ**と**後処理のベースライン**。初回コスト最小。 |
| **オプション** | **Qwen** 等（小型〜中型 Instruct） | **品質・多言語・後処理の安定**を求めるユーザー向け。ダウンロード後に選択可能。 |

**避けたいこと**：旧世代の**英語偏重・評価不明な超小型**だけをオプションの唯一の選択肢にする（学習言語によって差が出る）。

---

## 4. 運用上のプラクティス

- **プロンプト**：学習言語・説明言語・難易度を**テンプレで毎回明示**し、**Gemma／Qwen 切替**でも同じ枠に載せる。  
- **後処理**：候補の **Kind** はスキーマに近い形式を指定し、**軽量モデルでのパース失敗**時はリトライまたは短いフォールバックを設計する。  
- **継続評価**：**既定モデル × オプションモデル × 学習言語**の短い回帰セットを用意する。
