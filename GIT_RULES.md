# Git コミットメッセージ・テンプレート

## 1. 基本フォーマット

```text
<type>(<scope>): <subject>

<body>

<footer>

```

## 2. 複数修正を一括で行う場合の構成例

複数の変更をまとめる際は、本文の箇条書きを積極的に活用します。

```text
<type>: <全体の要約、または主要な修正内容>

- <修正内容1> (type)
- <修正内容2> (type)
- <修正内容3> (type)

Refs: #Issue番号

```

---

## 3. 各項目の入力ルール

### ① Type（種別）

そのコミットがどのような性質のものかを選択します。

| Type | 内容 |
| --- | --- |
| **feature** | 新機能の追加 |
| **fix** | バグ修正 |
| **docs** | ドキュメント（READMEなど）の修正 |
| **style** | コードの意味に影響しない修正（整形、セミコロンなど） |
| **refactor** | リファクタリング（機能追加もバグ修正も行わないコード整理） |
| **performance** | パフォーマンス改善 |
| **test** | テストコードの追加・修正 |
| **chore** | ビルド設定やライブラリ管理などの雑用 |

### ② Subject（要約）

* 1行目。50文字以内を推奨。
* 句読点（。）は不要。
* 複数の修正がある場合は、共通する目的（例: `update: ユーザー設定画面の調整`）を書く。

### ③ Body（詳細）

* なぜこの変更を行ったのか、背景や理由を記述。
* 複数の修正がある場合は、`-`（ハイフン）を使って箇条書きにする。

### ④ Footer（備考）

* 関連するIssueやプルリクエスト番号（例: `Fixes #12`）
* **BREAKING CHANGE**: 重大な仕様変更がある場合に明記。

---

## 4. 記述例（コピー用）

### ケースA：単一の修正

```text
fix: ログインボタンが二重送信される問題を修正

画面遷移中にボタンを非活性化することで、
APIへの重複リクエストを防止するように変更しました。

```

### ケースB：複数の修正をまとめる場合

```text
chore: プロジェクト構成の整理とREADMEの更新

- 不要なアイコン画像の削除 (chore)
- READMEに環境構築の手順を追記 (docs)
- typoの修正 (style)

```
